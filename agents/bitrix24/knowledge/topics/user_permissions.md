# Права пользователей Bitrix24 для AI-субагента

Этот topic читать перед любым Bitrix write-действием: созданием, изменением,
закрытием задач, комментариями от имени пользователя, заявками, CRM-записями и
другими действиями, которые меняют состояние портала.

Факты о текущем пользователе брать из `permission_context.bitrix_current_user_profile`
или read-only tool `current_user_profile`. Этот topic описывает правила
интерпретации этих фактов.

## Базовое правило

Если профиль пользователя из Bitrix недоступен, не угадывать права. Вернуть
`needs_clarification` или `needs_human` и попросить авторизоваться/обратиться к
пользователю с правами.

Если пользователь не активен (`active=false`) или тип пользователя не похож на
обычного сотрудника, write-действия не готовить.

Любое write-действие после проверки прав всё равно требует подтверждения в чате
и проходит backend guardrails/OAuth.

## Администраторы и директор

Пользователь считается администратором для AI-субагента, если Bitrix-профиль
явно содержит `is_admin=true` / `IS_ADMIN=true`, либо пользователь входит в
отдельно согласованный административный список в конфигурации AI-server.

Администратор может готовить следующие действия:
- создавать задачи в любых доступных проектах;
- готовить закрытие задач;
- готовить изменения задач и CRM, если соответствующий tool разрешён;
- создавать заявки и внутренние комментарии.

Если администратор просит опасное, массовое или необратимое действие, не
выполнять автоматически: подготовить черновик и запросить явное подтверждение.

## Монтажники

Монтажник определяется по одному из признаков в Bitrix-профиле:
- должность / `work_position` содержит `монтаж`;
- отдел/подразделение в `department_ids` соответствует монтажному отделу;
- в профиле есть другой явно согласованный признак монтажной роли.

MVP: точные ID монтажных отделов нужно заполнить после проверки структуры
Bitrix24. До заполнения ID использовать текстовые признаки должности и не
угадывать отдел по одному только имени пользователя.

Монтажник может:
- искать свои задачи и задачи своего проекта;
- добавлять результат выполнения по своей задаче;
- просить закрыть свою задачу через `task_closure`, если передан результат;
- создавать задачу/заявку в проекте `Задачи от сотрудников`, если такая
  возможность включена в конфигурации.

Монтажник не должен через AI-субагента:
- изменять чужие задачи без явного разрешения администратора/руководителя;
- закрывать чужие задачи;
- менять сроки, ответственного или проект у чужих задач;
- выполнять CRM-write действия.

## Обычные сотрудники с ограниченным созданием задач

Если пользователь не администратор, но входит в ограниченное правило создания
задач, он может готовить только `task_create_draft` в проекте
`Задачи от сотрудников`.

Если такой пользователь не указал проект, считать проектом по умолчанию
`Задачи от сотрудников`.

Если такой пользователь просит создать задачу в другом проекте, изменить задачу
или закрыть чужую задачу, не вызывать write-tool. Ответить, что для этого нужно
подтверждение/действие пользователя с расширенными правами.

## Неизвестная роль

Если по Bitrix-профилю и этому topic нельзя уверенно определить роль
пользователя, считать пользователя read-only для write-действий.

Read-only пользователь может искать и читать разрешённые ему данные, но не
должен готовить write-tools. Для записи предложить обратиться к администратору,
директору или ответственному сотруднику.
