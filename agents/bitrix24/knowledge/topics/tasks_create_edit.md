# Задачи: создание и редактирование

Читать этот раздел перед созданием, изменением, шаблонизацией или подготовкой `.docx`-шаблона задачи, если задача создаётся в рамках проекта/рабочей группы.

## Оглавление

- Область применения
- Создание проектной задачи
- Поля и возможности проектной задачи
- Чек-лист проектной задачи

## Область применения

Все правила из этого файла применять только к задачам, создаваемым или редактируемым в рамках какого-нибудь проекта/рабочей группы.

Если пользователь просит создать обычную задачу без проекта, не применять этот проектный шаблон, не требовать `GROUP_ID` и не навязывать проектные поля.

Если пользователь явно просит создать задачу в проекте, рабочей группе или называет конкретный проект, применять правила ниже.

## Создание проектной задачи

Создание и изменение задачи требуют подтверждения пользователя.

Обычные сотрудники с ограниченным правом могут создавать задачи через помощника
только в проекте `Задачи от сотрудников`. Если такой пользователь просит создать
задачу в другом проекте или изменить существующую задачу, честно сказать, что
это право ему не выдано.

Если обычный сотрудник с ограниченным правом просит создать задачу и не называет
проект, считать проектом по умолчанию `Задачи от сотрудников` и передавать его
ID в `GROUP_ID`.

Перед созданием проектной задачи:
- сначала прочитать topic `projects_crm`;
- найти проект/рабочую группу по названию;
- если найден один точный проект - передать его ID в `GROUP_ID`;
- если найдено несколько похожих проектов - показать варианты и попросить выбрать;
- если проект не найден - честно сказать, что проект не найден, и предложить уточнить название проекта.

Если проект не указан, не применять этот шаблон, не спрашивать проект автоматически и не передавать `GROUP_ID`.

Правило текущего tool-контракта `task_create_draft`:
- ответственный передаётся как `responsible_self` (если "на меня"/"себе"), явный
  `responsible_id` или `responsible_query` (ФИО/фамилия) — backend сам резолвит
  имя через Bitrix;
- проект/группа передаётся как явный `group_id` или `project_query` (название) —
  backend сам резолвит название через Bitrix;
- если backend не нашёл проект/пользователя однозначно или Bitrix API недоступен,
  он вернёт ошибку резолюции — в этом случае верни `needs_clarification`, а не
  угадывай ID. Отдельных read-only тулов `resolve_user`/`resolve_project` для
  прямого вызова из LLM не существует — резолюция происходит только внутри
  `task_create_draft`.

## Поля и возможности проектной задачи

Использовать при создании проектных задач и при подготовке `.docx`-шаблонов проектных задач.

Основные поля:

  а) Обязательные (если не указаны - запрашивать)
    - название / `TITLE`: если не указан, но есть описание - сделай сам кратко по смыслу из описания;
    - описание / `DESCRIPTION`: Если в описании больше одного действия для исполнителя - разбивать на пронумерованные шаги;
    - ответственный / `RESPONSIBLE_ID` если указано "я сам" или "на меня" - ставить `current_user.id`;
    - проект/группа / `GROUP_ID`: ID найденного проекта/рабочей группы;

  б) Необязательные (если не указаны - модель сама решает по правилам ниже или игнорирует)
    - соисполнители / `ACCOMPLICES`;
    - постановщик / `CREATED_BY`: всегда `current_user.id`; не подставлять другого пользователя и не пытаться назначать постановщиком администратора/webhook-пользователя;
    - наблюдатели / `AUDITORS`;
    - родительская задача / `PARENT_ID`;
    - приоритет / `PRIORITY`: дефолтное значение - "обычный";
    - крайний срок / `DEADLINE`: дефолтное значение - три рабочих дня от даты создания, время 19:00, с учётом выходных и настроенных праздничных дней;
    - плановая дата начала / `START_DATE_PLAN`: по дефолту равна дате создания;
    - плановая дата окончания / `END_DATE_PLAN`: по дефолту равна дедлайну;
    - разрешение менять срок / `ALLOW_CHANGE_DEADLINE`: по дефолту разрешено.

Сроки и планирование:
- если пользователь явно просит задачу без срока (`без срока`, `бессрочно`, `срок не нужен`), не передавать `DEADLINE`; можно передать служебный флаг `NO_DEADLINE: true`, чтобы backend безопасно сохранил отсутствие срока;
- плановая длительность / `DURATION_PLAN`;
- единица длительности / `DURATION_TYPE`;

Файлы и связи:
- вложения / `UF_TASK_WEBDAV_FILES`;
- связи CRM / `UF_CRM_TASK`;
- связанные задачи;
- связанные проекты;
- связанные документы.

Дополнительно:
- комментарии при постановке;
- повторяемость;
- учёт времени / `ALLOW_TIME_TRACKING`, `TIME_ESTIMATE`;
- теги;
- напоминания и уведомления;
- права и доступ;
- пользовательские поля `UF_*`;
- требования к результату перед завершением.

## Чек-лист проектной задачи

В шаблонах проектных задач учитывать:
- пункты чек-листа;
- ответственного за пункт;
- порядок;
- вложенность;
- завершённость;
- комментарии к пунктам;
- сроки по пунктам, если нужны.
