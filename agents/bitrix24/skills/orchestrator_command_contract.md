# Контракт точных команд оркестратора для Bitrix24

## Ответственность

Оркестратор сам определяет пользовательское намерение, выбирает один зарегистрированный Bitrix tool и заполняет его параметры. Если tool объявлен с `structured_command=true`, нельзя передавать специалисту свободный текст для повторного смыслового разбора. Неоднозначность обрабатывает оркестратор через `CLARIFICATION_REQUIRED`.

Bitrix проверяет версию контракта, JSON-схему, текущего пользователя, права, OAuth и состояние черновика, затем выполняет точную команду один раз.

## Поиск складов

- `найди`, `покажи`, `поищи`, `выведи`, `ищи` — синонимы; результат определяют объект и ограничения, а не глагол.
- Список складов: `bitrix_warehouse_search(query="все", list_all=true, include_products=false, limit=10)`. `list_all=true` обязателен: слово `все` не является названием склада.
- Один склад без другого объекта: карточка склада и первые 10 товаров — `include_products=true, product_limit=10, product_offset=0`.
- Товар на указанном складе: использовать только tool/capability `bitrix_warehouse_search`, добавить `product_query`, оставить `include_products=true`. Названия операций матрицы (`product_at_warehouse`, `warehouse_contents`) не являются capability.
- Явное `по складам` / `по всем складам`: отдельная складская команда на каждый найденный склад; результаты группируются.
- Адрес/местонахождение склада: `include_products=false`.
- `следующая`, `ещё`, `дальше`: тот же склад и фильтр, следующий `product_offset`.
- Если источник товара не назван и возможны разные системы, уточнить источник.

## Поиск задач

- `мои задачи`, `задачи`, `задачи на мне/на меня` без другого сотрудника: `bitrix_my_tasks`, текущий отправитель, активные статусы, 10 записей.
- `я исполнитель/ответственный`: `bitrix_task_search(scope="responsible")`.
- `поставленные мной`: `bitrix_task_search(scope="created_by")`.
- Только явное `все задачи`: `scope="all"`.
- ID, название, срок, просрочка, проект, комментарий или статус: `bitrix_task_search` с соответствующими параметрами.
- Если текущий пользователь не определён, личный запрос нельзя расширять до всех задач.

## Создание задачи

- Использовать `task_create_draft`; прямой `tasks.task.add` запрещён.
- Оркестратор заполняет `title`, ответственного, проект, срок и описание.
- Текущий пользователь: `responsible_self=true`.
- Ответственный по имени требует однозначного numeric ID; при нескольких совпадениях уточнить человека.
- Если срок не назван, не спрашивать: backend применит три рабочих дня, 19:00 МСК.
- `no_deadline=true` только при явном требовании задачи без срока.
- Запись выполняется только через `task_create_confirm` после точного подтверждения нужной ветки.

## Календарь и проекты

- Напоминание/событие: `calendar_event_draft`; без времени backend использует 12:00 МСК.
- Без явно названных участников событие создаётся только текущему пользователю.
- Проект: `project_create_draft`; перед созданием исключить однозначный существующий проект.
- Подтверждение и отмена используют соответствующие `*_confirm` / `*_discard` tools только для активного черновика текущей номерной ветки.

## Четырёхблочное закрытие задачи

1. Однозначно определить задачу и вызвать `task_close_draft`, а не прямые методы закрытия.
2. Блок 1 — пункты работы и только их статусы: выполнено / не выполнено / не подтверждено.
3. Блок 2 — только оборудование и расходники.
4. Блок 3 — общий статус и причины частичного/невыполненного результата.
5. Блок 4 — только дополнительная информация.
6. Не переносить текст между блоками. Неизвестное сохранять в `missing_fields` / `unconfirmed_items`, а не выдумывать.
7. Пустое описание исходной задачи: `source_task_description_empty=true`, результат пользователя — в `completion_summary`.
8. Уже закрытая задача: `already_closed=true`; подтверждение сохраняет AI-отчёт без повторного закрытия.
9. `task_close_confirm` допустим только после точного подтверждения соответствующего черновика.

## Безопасность

- Оркестратор планирует действие, но не объявляет право подтверждённым самостоятельно.
- Bitrix policy/OAuth/ACL являются обязательным окончательным барьером.
- Ошибка версии, схемы, доступа или состояния черновика останавливает выполнение; специалист не исправляет смысл команды самостоятельно.
- Автономные webhook/worker-процессы контроля закрытия не переводятся в разговорный оркестратор.

## Default project for a new task

- If no project is named, use the personal project of the resolved responsible user, not necessarily the sender.
- If no responsible user is named, the sender is responsible and the sender's personal project is used.
- A named project overrides the personal-project default.
- A named responsible user must be resolved to one exact numeric ID and label; ambiguity is clarified instead of guessing.
- A missing default personal project is prepared as an open and visible project owned by the responsible user, then the linked task draft is prepared.
