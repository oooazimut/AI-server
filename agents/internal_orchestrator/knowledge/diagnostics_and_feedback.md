# Глава: диагностика ошибок и feedback

Эта глава описывает, как Переговорщик должен относиться к ошибкам выполнения, пользовательским оценкам и будущему Diagnostic Agent.

## Краткий индекс

1. Роль Diagnostic Agent.
2. Что считается диагностическим событием.
3. Что должен сохранять контур диагностики.
4. Что Переговорщик не должен делать.
5. Как использовать результат диагностики.

## 1. Роль Diagnostic Agent

Diagnostic Agent - отдельный агент анализа ошибок.

Он не участвует в обычной цепочке ответа пользователю:

```text
Пользователь -> Оркестратор -> Bitrix24 -> Оркестратор -> пользователь
```

Его контур другой:

```text
feedback / trace / learning event -> Diagnostic Agent -> рекомендация исправления
```

Diagnostic Agent может знать об Оркестраторе, Bitrix Agent и их правилах как об объектах анализа. Боевые агенты не должны знать о Diagnostic Agent как о партнере обычного диалога.

## 2. Что считается диагностическим событием

Диагностическим событием считается:

- пользователь отметил задачу как невыполненную;
- пользователь поставил низкую оценку;
- ответ содержит ошибку специалиста или tool;
- Оркестратор выбрал неправильного специалиста;
- нужное правило не было подгружено;
- ответ был получен, но оказался неполным или бесполезным.

## 3. Что должен сохранять контур диагностики

Для будущего разбора нужно сохранять:

- исходный запрос пользователя;
- финальный ответ;
- выбранные tool calls;
- результаты tools;
- actions_taken;
- model_usage;
- загруженные правила и причину загрузки;
- feedback пользователя;
- trace_id, когда TraceRecorder будет добавлен.

## 4. Что Переговорщик не должен делать

Переговорщик не должен:

- вызывать Diagnostic Agent как обычного специалиста для ответа пользователю;
- скрывать ошибку, если tool или специалист вернул ошибку;
- менять правила на лету без подтверждения человека;
- придумывать причину сбоя без trace, actions и feedback.

## 5. Как использовать результат диагностики

Результат диагностики должен помогать разработке:

- указать вероятный этап сбоя;
- предложить, что исправить: правило, skill, tool, код или тест;
- предложить regression test;
- связать исправление с повторяющимися incidents.

Исправления правил и кода выполняются отдельно, после решения человека.
