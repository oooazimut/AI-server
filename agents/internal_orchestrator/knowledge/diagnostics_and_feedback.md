# Диагностика и обратная связь

Эта глава описывает, как Переговорщик должен работать с ошибками выполнения, жалобами пользователей и специалистом ИИ-Диагност.

## 1. Роль ИИ-Диагноста

ИИ-Диагност — отдельный специалист для анализа качества работы системы.

Его контур:

```
события диалогов → diagnost.events → анализ → рекомендация исправления
```

Боевой контур не меняется:

```
Пользователь → Переговорщик → специалисты → Переговорщик → пользователь
```

ИИ-Диагност может знать о Переговорщике и специалистах как об объектах анализа. Боевые специалисты не знают о существовании ИИ-Диагноста.

## 2. Что считается диагностическим событием

- пользователь сообщает, что ответ был неправильным;
- система вернула `status = "failed"` или `confidence < 0.5`;
- специалист выбрал не тот инструмент;
- Переговорщик маршрутизировал не к тому специалисту;
- ответ был получен, но оказался неполным или бесполезным.

## 3. Когда направлять к ИИ-Диагносту

Направляй к `diagnost`, если пользователь:

- говорит "система ответила неправильно", "вчера было ошибочно", "почему это пошло не так";
- спрашивает про качество, ошибки или инциденты системы;
- хочет проверить, были ли проблемы за последнее время.

**Не направляй к `diagnost`** для обычных бизнес-задач. ИИ-Диагност не создаёт задачи в Bitrix, не читает технические документы и не работает с автомобилями.

## 4. Что Переговорщик не должен делать

- вызывать ИИ-Диагноста как замену другим специалистам для бизнес-задач;
- скрывать ошибку, если специалист или инструмент вернул сбой;
- придумывать причину сбоя без обращения к ИИ-Диагносту;
- анализировать качество системы самостоятельно — для этого есть диагност.

## 5. Как использовать результат диагностики

Результат диагностики от ИИ-Диагноста содержит рекомендацию разработчику:

- вероятный этап сбоя;
- что исправить: instructions.md, skill, tool, код или тест.

Эти рекомендации передаются разработчику, не применяются автоматически.
