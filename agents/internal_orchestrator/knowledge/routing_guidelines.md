# Глава: правила маршрутизации запросов

Эта глава помогает Переговорщику выбрать специалиста или понять, что специалист не нужен.

## Краткий индекс

1. Текущий рабочий контур.
2. Таблица выбора специалиста.
3. Граничные случаи.
4. Когда вызывать нескольких специалистов.
5. Когда отвечать самому.
6. Как расширять маршрутизацию новыми агентами.

## 1. Текущий рабочий контур

В рамках текущего этапа основные специалисты для пользовательских задач - `call_bitrix24` и `call_secure_org_data`.

`call_bitrix24` отвечает за Bitrix24: задачи, проекты, CRM, диск Bitrix, поиск по порталу, уведомления, сообщения, складские/товарные запросы, каталог и номенклатуру.

`call_secure_org_data` отвечает за локальную базу организации: объекты, оборудование, инструкции, настройки, сетевые параметры, пути к локальным документам, логины, пароли и учетные данные. Агент работает read-only и использует только уже заданные метки доступа.

Diagnostic Agent нужен для анализа ошибок, feedback и trace. Он не должен вызываться как обычный специалист для ответа пользователю.

`call_pto` и `call_logistics` существуют в проекте, но в этом этапе их правила и сценарии не расширяются.

| Тема запроса | Инструмент |

| Задачи Bitrix24: создание, комментарии, смена статуса, дедлайны | `call_bitrix24` |
| Проекты, CRM и поиск по порталу Bitrix24 | `call_bitrix24` |
| Диск Bitrix24: поиск файлов, загрузка документов, прикрепление документов | `call_bitrix24` |
| Уведомления и сообщения в Bitrix24-чатах | `call_bitrix24` |
| Склад, наличие товара, товарный поиск, каталог и номенклатура | `call_bitrix24` |
| Локальные инструкции по оборудованию, объектам и настройкам организации | `call_secure_org_data` |
| Поиск в локальной базе организации, включая `C:\Настройки, Инструкции-` | `call_secure_org_data` |
| Сетевые параметры, IP-адреса, логины, пароли и учетные данные организации | `call_secure_org_data` |
| Локальные документы, файлы и пути, не относящиеся явно к Bitrix24 | `call_secure_org_data` |
| Низкая оценка, "не выполнено", разбор сбоя, trace, feedback | не обычный handoff; данные идут в diagnostic-контур |
| ПТО или логистика | вне активного этапа работ; не расширять правила в этой ветке |

## 3. Граничные случаи

**"Покажи документ к задаче"**  
Сначала нужен `call_bitrix24`, чтобы найти задачу или документ в Bitrix24. Если затем требуется смысловой анализ документа, это уже не основной сценарий текущего этапа.

**"Найди инструкцию TL-WR820N" / "Найди настройки роутера"**  
Используй `call_secure_org_data`, потому что это поиск локальной инструкции, оборудования или настроек организации. Не задавай уточнение, если в запросе уже есть модель, название оборудования, IP, объект или технический термин.

**"Найди датчик коленвала на складе"**  
Если пользователь явно говорит про склад, товар, наличие, каталог или номенклатуру, вызывай `call_bitrix24` без уточнения на первом шаге, потому что складской/товарный поиск относится к бизнес-системам и каталогу. Если запрос говорит про инструкцию, настройки, объект или локальный документ по этому датчику, используй `call_secure_org_data`.

**"Что нового в портале?"**  
Используй `call_bitrix24`, потому что этот специалист работает с задачами, диском и поиском по порталу.

**"Как дела?" / "Спасибо"**  
Специалист не нужен. Отвечай сам через `tool_calls=[{"name":"none"}]`.

**Неоднозначный запрос**  
Задай один конкретный уточняющий вопрос. Не угадывай домен. Но если запрос содержит явный технический маркер локальной базы организации, например модель оборудования, IP, логин, пароль, объект, инструкция или настройки, сначала вызывай `call_secure_org_data`.

**"Ответ неверный" / "задача не выполнена"**  
Это не обычная пользовательская задача для Bitrix. Нужно сохранить feedback/trace для диагностического контура и не пытаться исправлять прошлый ответ без данных.

## 4. Когда вызывать нескольких специалистов

Вызывай нескольких специалистов, если запрос явно касается нескольких доменов.

Если результат одного специалиста нужен другому, действуй последовательно.

Если части запроса независимы, можно вызвать нескольких специалистов в одном цикле.

Не вызывай одного и того же специалиста дважды в одном цикле.

В текущем этапе не добавляй новые многоагентные сценарии для `pto` и `logistics`.

## 5. Когда отвечать самому

Переговорщик может ответить сам, если:

- это приветствие или благодарность;
- это общий вопрос о системе, боте или возможностях;
- запрос уже полностью обработан предыдущими `tool_results`;
- нет необходимости вызывать инструмент.

## 6. Как расширять маршрутизацию новыми агентами

При добавлении нового специалиста не переписывай старые правила.

Для нового агента нужно добавить отдельную routing-подсказку:

- какие темы относятся к новому агенту;
- какой `call_<agent_id>` использовать;
- какие граничные случаи пересекаются с уже существующими специалистами;
- когда новый агент основной, а когда дополнительный.

Старые правила меняются только если новый агент действительно забирает часть уже описанной ответственности.
