# Глава: анализ событий диалогов

## Назначение

Используй эту главу, когда нужно разобрать конкретный диалог оркестратора по данным из `diagnost.events`.

## Что хранится в событии

| Поле | Описание |
|---|---|
| `event_id` | task_id оркестратора (связывает с инцидентом) |
| `request` | исходный запрос пользователя |
| `response` | финальный ответ оркестратора |
| `status` | `completed`, `failed`, `needs_human`, `handoff` |
| `confidence` | уверенность 0.0–1.0 |
| `handoff_to` | специалисты, которым передана задача |
| `actions` | tool-вызовы (AgentTool) |
| `model_usage` | токены LLM |
| `channel` | канал (bitrix24_chat и т.д.) |

## Этапы анализа

1. **Запрос** — правильно ли понят вопрос пользователя.
2. **Маршрутизация** — правильный ли специалист в `handoff_to`.
3. **Tool-вызовы** — какие инструменты вызваны, с правильными ли аргументами.
4. **Финальный ответ** — соответствует ли ожиданию пользователя.
5. **Confidence и status** — признаки автоматического инцидента.

## Что считать сбоем

- Переговорщик выбрал не того специалиста.
- Нужный специалист не был вызван.
- Tool вернул ошибку, но финальный ответ это скрыл.
- Tool вернул данные, но агент неправильно их интерпретировал.
- Финальный ответ потерял важную информацию.
- `status = "failed"` или `confidence < 0.5`.

## Правила осторожности

- Не делай вывод "tool ошибся", если `actions` пустые или отсутствуют.
- Если `confidence` низкий, но ответ кажется правильным — уточни у человека.
- Если событие не найдено, сообщи об этом явно и попроси уточнить ключевые слова.
