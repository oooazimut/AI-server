# Roadmap: реальные интеграционные тесты AI-server

Документ фиксирует этап `feature/real-integration-tests`: подготовку максимально близких к боевой работе тестов перед переносом проекта на сервер.

Главная целевая цепочка:

```text
Пользователь в Bitrix-мессенджере
-> Bitrix Bot
-> /bitrix/events
-> webhook queue
-> internal_orchestrator / Переговорщик
-> нужный специалист
-> internal_orchestrator / Переговорщик
-> ответ обратно в чат Bitrix
```

Старый `/orchestrator/test` остается только как вспомогательный API-runner для быстрой проверки логики без Bitrix-мессенджера.

## 2. Публичный URL для локального теста

Статус: выполнено.

Цель: дать Bitrix24 внешний HTTPS адрес, по которому он сможет отправлять события бота на локальный AI-server.

Фактическое решение:
- используется домен `kitvideovpn.ru`;
- домен указывает на публичный IP `83.221.222.37`;
- Keenetic пробрасывает внешний TCP `1695` на dev-ПК `192.168.0.64:1695`;
- AI-server слушает `0.0.0.0:1695` с TLS-сертификатом из `var/certs/`;
- рабочий внешний health-check: `https://kitvideovpn.ru:1695/health`.

Для Bitrix Bot нужен URL вида:

```text
https://kitvideovpn.ru:1695/bitrix/events?secret=<secret>
```

Ограничения:
- секрет хранить только в `.env.local`;
- не выводить полный URL с реальным `secret` в чат и логи;
- сертификаты и ключи держать только в `var/certs/` или другой git-ignored runtime-папке;
- после перезагрузки ПК или остановки процесса заново запускать dev HTTPS-сервер на `1695`.

## 3. Bitrix Bot в мессенджере

Статус: выполнено для dev-бота, prod-бот не изменялся.

Цель: подключить реального бота в Bitrix-мессенджере.

Сделано:
- создан/подключен отдельный dev-бот `AI dev`;
- prod-бот `AI-помощник` не изменялся;
- текущий dev bot id: `231`;
- `BITRIX_REST_WEBHOOK_URL`, `BITRIX_BOT_ID`, `BITRIX_BOT_TOKEN`, `WEBHOOK_SECRET` заданы в `.env.local`;
- Bitrix отправляет событие `ONIMBOTMESSAGEADD` на `/bitrix/events`;
- `/bitrix/webhook-events/status` видит входящие события, очередь обрабатывает их без ошибок;
- подтверждено, что обычные сообщения в групповой dev-чат без реального упоминания бота не запускают нужную обработку, а прямой диалог с ботом работает стабильно.

В проекте уже есть основа:
- `/bitrix/events` принимает webhook;
- `WebhookEventQueue` кладет события в `var/webhook_event_queue.sqlite`;
- `chat_parser.py` превращает сообщение Bitrix в `AgentTask`;
- worker отправляет задачу в `orchestrator`;
- `BitrixChatChannel` отправляет ответ обратно в диалог Bitrix.

## 6. Trace, feedback и Diagnostic Agent

Статус: базовая цепочка в реальном Bitrix-чате проверена, нужно расширить детализацию.

Подробный roadmap по структуре incident, diagnostic_report, классификации, группировке и отчетам вынесен в `docs/11-diagnostic-error-roadmap.md`.

На каждый тест должно быть видно:
- `trace_id`;
- входящее сообщение пользователя;
- решение Переговорщика;
- вызванный специалист;
- tool calls специалиста;
- финальный ответ;
- отправка сообщения обратно в Bitrix.

Для плохого ответа:
- человек ставит плохую оценку или помечает задачу как невыполненную;
- создается feedback;
- создается incident;
- Diagnostic Agent кратко показывает причину и предлагает исправление.

Нужно реализовать обязательный feedback-loop после каждого ответа бота:
- каждый ответ в Bitrix-чате должен завершаться коротким вопросом: оценить ответ по 10-балльной шкале и, если что-то не так, написать пояснение;
- следующий ответ пользователя, похожий на оценку (`7`, `3/10`, `8 все ок`, `2 не нашел товар`), должен распознаваться как feedback к последнему ответу бота в этом диалоге, а не как новый обычный запрос;
- feedback должен сохраняться через `/learning/feedback` с `rating_scale=10`, `comment`, `channel=bitrix24_chat`, `user_id` и ссылкой на последний `learning_event_id`;
- низкая оценка или проблемное пояснение должны создавать `incident`;
- если создан incident, диагностический контур должен вызывать `/learning/diagnose` и сохранять `diagnostic_report`, связанный с исходным ответом, feedback и incident;
- если пользователь написал новый содержательный запрос вместо оценки, бот должен обработать его как обычный запрос, не блокируя диалог ожиданием feedback.

Что уже увидели на реальном тесте:
- learning event фиксирует `source=bitrix24_chat`;
- видно, что `internal_orchestrator` вызвал `bitrix24`;
- текущий diagnostic trace сохраняет решение оркестратора, но не полностью раскрывает внутренние actions specialist-а.

Нужно улучшить:
- добавить в trace/learning event краткую сводку actions specialist-а, хотя бы `bitrix_current_user_profile_status` и имя раннего ответа;
- добавить состояние ожидания оценки в dialog state: последний `learning_event_id`, `trace_id`, текст ответа, время отправки и признак, что следующий короткий rating-like ответ нужно трактовать как feedback;
- добавить parser пользовательской оценки по 10-балльной шкале и комментария, включая форматы `7`, `7/10`, `оценка 7`, `7 все ок`, `3 не помогло`;
- после записи feedback и diagnostic_report фиксировать в trace событие `human_feedback_received` и связь с `diagnostic_report_id`;
- отдельно проверять, что все новые real-chat сценарии проходят через `internal_orchestrator`.

## 7. Правка логики по результатам прогонов

Статус: ожидает реальных прогонов через Bitrix-бота.

Править в таком порядке:
- правила маршрутизации Переговорщика;
- `skill_index.yaml` и skills Bitrix Agent;
- правила складского поиска;
- формулировку коротких ответов;
- trace и diagnostic-поля;
- regression tests.

Уже найденный пример:
- запрос `найди датчик коленвала на складе гараж` не должен требовать лишний артикул, если товар и склад уже названы;
- ожидаемое поведение: сразу искать остатки и вернуть результат или `0 шт`.

## 8. Write-тесты

Статус: отложено.

Write-тесты делать только после отдельного подтверждения пользователя.

Ограничения:
- только тестовая группа или тестовый проект Bitrix;
- сначала подтверждение человека;
- отдельные сценарии на создание задачи, комментарий и изменение статуса;
- никакой массовой записи и удаления данных.

## 9. Ближайший план

Статус: следующий рабочий блок после успешного dev-бот smoke test.

Цель: расширить проверку от одного успешного Bitrix smoke test к набору read-only сценариев всех агентов через `internal_orchestrator`.

Делать дальше:
- держать dev AI-server запущенным на `https://kitvideovpn.ru:1695`;
- для каждого нового теста проверять `/bitrix/webhook-events/status` и learning event без вывода секретов;
- прогнать read-only сценарий `покажи мои открытые задачи`;
- прогнать сценарий поиска задачи по тексту;
- прогнать сценарий поиска документа/файла через PTO/document tools, если индекс и доступы готовы;
- прогнать сценарий Secure Org Data только с безопасным тестовым запросом и без раскрытия protected data;
- прогнать логистический/складской сценарий `найди датчик коленвала на складе гараж`;
- проверить ошибочный запрос и связку feedback -> Diagnostic Agent;
- реализовать и проверить real-chat feedback-loop: бот спрашивает оценку после ответа, пользователь отправляет оценку с пояснением, создаются feedback, incident и `diagnostic_report`;
- после каждого прогона фиксировать: входящее событие, решение `internal_orchestrator`, вызванный specialist, tool calls, финальный ответ, отправку обратно в Bitrix.

Пройденная первая проверка:
- `кто я в Битрикс` в прямом диалоге `AI dev`;
- результат: бот вернул профиль пользователя Bitrix24;
- замечание: после перезапуска dev-сервера старый процесс был заменен, и актуальный код начал отвечать корректно.

Важно:
- текущий запуск на `1695` является dev-запуском на ПК, не prod-службой;
- если процесс остановится или ПК перезагрузится, его нужно поднять заново;
- перед write-сценариями отдельно запросить подтверждение пользователя.

## Правила безопасности

- Не коммитить `.env.local`.
- Не выводить webhook URL, OAuth tokens, DeepSeek key и другие секреты.
- Не коммитить `outputs/` без отдельной команды пользователя.
- Не пушить в GitHub без отдельной команды.
- Не писать в основную ветку.
- Protected/secret данные Secure Org Data не раскрывать без согласованных правил доступа.
