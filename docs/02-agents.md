# Агенты

## Агент как пакет

Специалист - это не один prompt. В проекте специалист описывается пакетом:

```text
manifest.yaml      # машинная карточка агента
instructions.md    # роль, границы, правила ответа
skills/*.md        # процедурные сценарии
knowledge/*.md     # доменные знания/RAG-источники
tools              # разрешенные инструменты через backend
policies           # правила допуска и подтверждений
automations        # фоновые процессы домена, если они есть
evals              # тестовые кейсы поведения
```

## Оркестратор

Оркестратор принимает сообщение от канала, выбирает специалиста через `Agent Registry`, передает ему `AgentTask`, получает `AgentResult` и формирует ответ пользователю.

Оркестратор знает версионированный каталог возможностей исполнителей, но не вызывает
низкоуровневый Bitrix REST напрямую.

## Битрикс24-специалист

Битрикс24-специалист выделен из старого автономного проекта `BitrixAIAgent`. Его спецификация находится в `agents/bitrix24`.

Он отвечает за:

- задачи и заявки;
- проекты и рабочие группы;
- CRM;
- поиск и доступ к документам/файлам портала как Bitrix-объектам;
- локальный индекс портала;
- подготовку черновиков Bitrix-действий.

Смысл запроса, параметры инструментов и итоговый формат ответа всегда определяет
Pro-оркестратор. ПТО и Картотека сейчас не являются подключёнными специалистами.

## Executor и operator-контуры

`logistics` — доступный оркестратору структурированный executor без LLM. Он публикует
версионированный реестр инструментов, принимает одну точную JSON-команду и не
интерпретирует пользовательский свободный текст. Его автоматизация по расписанию
остаётся детерминированным фоновым процессом.

`diagnost` — технический operator-контур без LLM, недоступный в пользовательской маршрутизации.

## Контракт вызова специалиста

```json
{
  "task_id": "...",
  "source": "internal_orchestrator",
  "user": { "id": "...", "channel": "bitrix24_chat" },
  "request": "...",
  "files": [],
  "context": {},
  "allowed_actions": [],
  "required_output_format": "structured_result"
}
```

## Ответ специалиста

```json
{
  "status": "completed | needs_clarification | needs_human | failed",
  "agent_id": "bitrix24",
  "answer": "...",
  "artifacts": [],
  "actions_taken": [],
  "actions_requiring_approval": [],
  "confidence": 0.8,
  "logs": []
}
```

## RAG и Skills

RAG отвечает за знания: регламенты, инструкции, документацию, историю и справочники.

Skills отвечают за процедуру: какие шаги выполнить, какие инструменты вызвать, где остановиться и когда запросить подтверждение.

В production нужны оба слоя.

## Автоматизации

Автоматизации - это не субагенты. Они описывают процессы, которые живут по
событиям или расписанию: очередь webhook-событий, индексация, контроль качества,
напоминания по просроченным задачам.

У специалиста может быть свой список автоматизаций в `manifest.yaml`, но
специалист не обязан запускать их из диалога. Он может пользоваться их результатом
или объяснять их состояние, а жизненный цикл worker-ов остаётся в backend runtime.
